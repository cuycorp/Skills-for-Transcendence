## 2. Product in one page

### Product name

**TokenScope: Collaborative LLM Cost Observatory**

### One-sentence pitch

TokenScope is a collaborative platform that helps engineering teams understand where their LLM spending comes from by collecting usage traces, calculating estimated cost, visualizing live analytics, and generating grounded optimization recommendations.

### Target user

A small engineering team operating an AI feature such as a customer-support chatbot, RAG assistant, document-analysis workflow, or agent.

### Their problem

The team sees an increasing LLM bill but cannot precisely answer:

- Which workflow is expensive?
- Which model consumes the most budget?
- Is the cost caused by large inputs, long outputs, retries, failures, or too much RAG context?
- What should we change first?
- How can we reduce cost without blindly degrading answer quality?

### Our school-project solution

A user can:

1. create an organization and a project;
2. generate a project API key;
3. send LLM usage events to a secured API;
4. observe cost, tokens, latency, and errors in a live dashboard;
5. inspect individual traces;
6. upload technical documents to a knowledge base;
7. ask an AI assistant grounded in that knowledge base for explanations;
8. request an explanation and recommendation for a selected expensive trace.

### Core promise

> TokenScope turns raw LLM telemetry into understandable engineering decisions.

### Scope statement

The project demonstrates **visibility, explanation, and safe recommendations**.

It does **not** claim to be a complete commercial LLM gateway or to automatically reduce every customer bill by a fixed percentage.

## 3. Product vocabulary

| Term | Meaning in TokenScope |
|---|---|
| LLM | Large Language Model, for example a model used to generate text. |
| Provider | Company/API supplying a model, such as OpenAI, Anthropic, or Google. |
| Model | A specific LLM selected for a request. |
| Token | A unit of text used by LLM providers for context limits and billing. |
| Input tokens | Tokens sent to the model: prompt, history, documents, tool definitions. |
| Output tokens | Tokens generated by the model as a response. |
| Trace | One recorded LLM execution event, including model, tokens, latency, status, and cost. |
| Workflow | A named business action, for example `support_answer` or `document_summary`. |
| Project | An isolated TokenScope workspace inside an organization. |
| API key | A secret used by a client application to send traces to a particular project. |
| RAG | Retrieval-Augmented Generation: retrieve relevant documents, then pass them to an LLM. |
| Embedding | Numeric representation of text used to search for semantically similar content. |
| Vector search | Search over embeddings to retrieve relevant document chunks. |
| Streaming | Delivering an LLM response progressively rather than waiting for the complete response. |

## 4. User story and first demo

### Primary user journey

```text
Sign up
  → Create organization
  → Create project
  → Generate API key
  → Send trace event
  → Dashboard updates live
  → Open expensive trace
  → Ask “Why is this expensive?”
  → Receive grounded explanation and recommendation
```

### Demo scenario

1. **Free Support** creates the project `Customer Support AI`.
2. It uses a generated API key to submit traces for its `support_answer` workflow.
3. Five minutes later, the dashboard shows that this workflow represents 62% of spend.
4. A user opens an expensive trace.
5. The assistant explains that the trace has unusually high input context and an unnecessarily long output.
6. The assistant recommends testing a lower `top-k` retrieval value and a lower response token limit.

This one scenario demonstrates the entire product coherently.

## 5. Scope: what we build versus what we deliberately postpone

### In scope for ft_transcendence

- Multi-user web application.
- Authentication with email/password.
- Organizations, members, projects, and role-aware permissions.
- API-key secured trace ingestion API.
- Token and estimated-cost calculation.
- Analytics dashboard with filters and export.
- Real-time dashboard updates with WebSockets.
- RAG system over uploaded technical documents.
- Streaming LLM assistant.
- Trace explanation and rule-based recommendation engine.
- Docker Compose deployment.

### Explicitly out of scope

- Transparent proxy for all LLM providers.
- Automatic provider/model routing in production.
- Automatic prompt rewriting.
- Automatic model switching.
- Self-hosted LLM inference.
- Microservices, Kafka, Kubernetes, or a custom vector database.
- Payments, billing, or commercial subscription management.
- Production compliance certifications.

**Reason:** a coherent, fully functional vertical slice is more valuable than ten incomplete advanced ideas.

## 7. Architecture: one modular monolith

### Architectural decision

We build **one modular monolith**, not microservices.

Why:

- We are a small team learning a new stack.
- The subject rewards complete working modules, not unnecessary distributed-system complexity.
- A modular monolith is easier to run, debug, test, and explain.
- Clear module boundaries still prepare the system for later extraction if needed.

### High-level system

```text
Browser
  │ HTTPS / WebSocket
  ▼
React + TypeScript frontend
  │ REST / WebSocket
  ▼
NestJS backend (modular monolith)
  ├── Identity & Authorization
  ├── Organizations & Projects
  ├── API Keys
  ├── Trace Ingestion
  ├── Cost Engine
  ├── Analytics
  ├── Realtime Gateway
  ├── Knowledge Base / RAG
  └── AI Assistant
  │
  ├── PostgreSQL + pgvector
  ├── File storage volume
  └── LLM / embedding provider API
```

### Technology stack

| Layer | Choice | Reason |
|---|---|---|
| Frontend | React + TypeScript + Vite | Clear ecosystem, component model, rapid iteration. |
| UI | Tailwind CSS | Fast, responsive, reusable styling. |
| Backend | NestJS + TypeScript | Modular architecture, validation, WebSockets, similar to structured backend patterns. |
| Database | PostgreSQL | Strong relations, analytics queries, reliable transactions. |
| ORM | Prisma | Typed schema, migrations, productive developer experience. |
| Vector search | pgvector | Keeps relational and vector data in one database. |
| Realtime | Socket.IO / WebSocket | Live dashboard updates and connection handling. |
| AI | One LLM provider + one embeddings API initially | Learn the fundamentals before multi-provider support. |
| Containers | Docker Compose | Single-command reproducible environment. |
| API docs | Swagger / OpenAPI | Required public API discoverability and evaluator-friendly documentation. |

## 8. Core data model

### Main entities

```text
User
  └── Membership ──> Organization
                        └── Project
                              ├── ApiKey
                              ├── Trace
                              ├── ModelPrice
                              ├── Document
                              │     └── DocumentChunk
                              └── Conversation
                                    └── Message
```

### Essential tables

| Table | Purpose |
|---|---|
| `users` | Authentication identity and profile. |
| `organizations` | Collaborative tenant/workspace. |
| `memberships` | Links users to organizations and roles. |
| `projects` | Isolated monitoring/RAG workspace. |
| `api_keys` | Hashed keys for trace-ingestion authorization. |
| `traces` | LLM execution telemetry: model, workflow, tokens, latency, cost, status. |
| `model_prices` | Pricing catalog used to estimate trace cost. |
| `documents` | Uploaded knowledge-base files. |
| `document_chunks` | Text chunks and embeddings for RAG retrieval. |
| `conversations` | Assistant conversation sessions. |
| `messages` | User/assistant messages with source metadata. |

### Non-negotiable data rules

- Every business record belongs to an organization or a project.
- Every database query is scoped by authorization context.
- API keys are never stored in plaintext after creation.
- User input is validated in both frontend and backend.
- Prices are versioned or timestamped; cost must be reproducible.

## 10. How we work

### Git workflow

```text
main        = stable and demoable
develop     = integration branch
feature/... = one focused feature branch
fix/...     = focused defect branch
```

Rules:

- No direct push to `main`.
- Every non-trivial change is made through a pull request.
- One reviewer minimum for important changes.
- A PR must link to an issue and explain what changed, how it was tested, and what remains limited.
- Small, meaningful commits only.

### Commit convention

```text
feat(traces): add authenticated trace ingestion endpoint
fix(auth): prevent cross-organization project access
docs(architecture): add API-key hashing decision
refactor(analytics): isolate daily cost aggregation query
test(rag): add retrieval relevance cases
```

### Meetings

- **Friday demo/weekly - between 11h - 14h Healthy check (Ad-hoc if needed):** every owner demonstrates working progress.
- **Ad hoc pairing:** whenever an implementation is blocked for more than 45 minutes.

### Definition of Ready

A task may enter development only when it has:

- a clear user outcome;
- acceptance criteria;
- API/data impact identified;
- known dependencies;
- a named owner;
- a reasonable test plan.

### Definition of Done

A task is done only when:

- feature behavior matches acceptance criteria;
- authorization and tenant scoping are considered;
- frontend/backend validation exists where applicable;
- tests pass;
- no browser-console errors;
- PR is reviewed;
- relevant documentation is updated;
- the owner can demo and explain it.

## 11. First vertical slice: the only goal before advanced AI

### Vertical Slice 0 — “Trace to dashboard”

```text
User signs up
  → creates organization
  → creates project
  → generates API key
  → sends one trace event
  → backend validates and stores it
  → cost is calculated
  → dashboard displays the trace and total spend
```

### Why this is first

It proves the actual product loop before RAG, streaming, advanced charts, or complex AI features.

If this slice does not work, the project does not yet have a product.

### Acceptance criteria

- Two users can belong to the same organization.
- A project belongs to exactly one organization.
- A project API key is created once and stored only as a hash.
- An authenticated request can create a trace.
- The backend rejects an invalid key, invalid project, invalid token count, or invalid payload.
- The cost is calculated from a model-price record.
- The dashboard shows total spend and the new trace.

## 18. Learning and delivery operating model

### 18.1 The rule: the lead stays one sprint ahead

TokenScope is not a project where the Technical Lead waits for the team to finish and then checks code.

Léon operates one sprint ahead of implementation:

```text
Sprint N:
  Team implements the current vertical slice.
  Léon studies and designs Sprint N+1.

Before Sprint N+1 starts:
  Léon has written the feature brief, API contract, acceptance criteria,
  relevant ADRs, data-model changes, risks, and demo scenario.
```

This is the right form of control. It prevents blind implementation while avoiding a bottleneck where one person writes every line of code.

### 18.2 Learning rules

1. Learn only what unlocks the next vertical slice.
2. No contributor starts an implementation before understanding the feature's user outcome and acceptance criteria.
3. Every new domain begins with a short team primer, a small spike, then a production feature.
4. The team studies one provider and one path first; no multi-provider abstraction before the single-provider path works.
5. The lead documents decisions before implementation, not after a bug or disagreement.
6. No one treats AI as magic: every output, trace, and recommendation must be inspectable.

### 18.3 Time allocation
Proposition:
```text
20–25% learning / spike work
60–65% implementation
10–15% tests, documentation, review
```

## 21. How a feature enters the team backlog

Every feature begins with a **Feature Brief** prepared by Team.

```markdown
# Feature: Project API Key Creation

## User outcome
An organization owner can create a project API key and copy it once.

## In scope
- Create key for a project.
- Show raw key only once.
- Store only its hash.
- Revoke a key.

## Out of scope
- Key rotation scheduling.
- Multiple permission scopes.

## API contract
POST /projects/:projectId/api-keys
DELETE /projects/:projectId/api-keys/:keyId

## Data impact
ApiKey(projectId, label, prefix, hash, createdAt, revokedAt)

## Security requirements
- Owner/admin only.
- API key plaintext never reappears after response.
- Key lookup uses a safe hash comparison.

## Acceptance criteria
- Valid owner can create and copy a key once.
- Non-member receives 403.
- Revoked key fails trace ingestion.
- No plaintext key is stored in database or logs.

## Test plan
- Controller/service integration tests.
- Authorization test.
- Revocation test.
- Manual UI demo.
```
